# EFW 编排外壳（项目级 alwaysApply 规则）

> **为什么放这里**：用户级 `~/.workbuddy/MEMORY.md` 的 EFW 纪律**只对主对话生效**，子 agent / 别的运行环境不读它。放进项目 `.workbuddy/rules/` 且 `alwaysApply: true`，才会**每会话、对每个读本项目的 agent（含子 agent）自动注入**，是软规则里触达面最广的一档。
>
> **用法**：把本文件复制到项目根的 `.workbuddy/rules/efw-shell.md`。

## ⛔ 实质任务 → 第一动作先加载 efw

**判据（满足任一即"实质任务"）**：① 产出文件/交付物（.pdf/.xlsx/.pptx/.docx/.html/代码/设计稿…）；② 需 ≥3 步；③ 预计 ≥15 次工具调用。
→ 纯问答 / 查一条资料 / 单步文本 / 闲聊**不算**，直接做，别套外壳。

一旦判定为实质任务，**你的第一个动作就是加载 `efw` 技能**（先于任何领域工具/脚本）。这是机械默认动作，**不等用户点名**。EFW 是"外壳"但载体是可选 skill——系统不会自动加载，必须你显式发起。

## 🚫 反"暗合"替代——以下都不算走了 EFW

| 你实际做了 | ≟ | EFW 要求的 | 判定 |
| --- | --- | --- | --- |
| 用 TaskCreate 拆了任务 | ≠ | 走了编排外壳 | ❌ |
| 自写校验脚本 / 肉眼看一眼 | ≠ | `efw-verify` 逐条结构化签收 | ❌ |
| 写条 memory 笔记 | ≠ | `efw-learn` 沉淀 | ❌ |
| "四步心里过了一遍" | ≠ | 实际加载执行 efw-* | ❌ |

**神似 ≠ 触发。** 四步必须真的走技能。

## 四步外壳

```
① 计划  →  ② 干活(交对口领域技能/原生工具)  →  ③ 验收(efw-verify)  →  ④ 沉淀(efw-learn)
```

- **①计划**：对口领域技能自带 plan/大纲的视为满足，不叠 `efw-plan-feature`（避免双规划）。
- **②干活**：交给最对口的领域技能；EFW 不跟它抢。
- **③验收（强制闸门，领域技能不豁免）**：对照验收标准**逐条勾选、给可核验清单**，不是抽查交差。
- **④沉淀**：≥15 调用或有可复用套路 → `efw-learn`（可复用流程存成新 skill）。

## 🚦 DoD 硬闸门

没跑 `efw-verify`（逐条对照验收标准签收）之前，**不许说"做完了 / 已完成"**。文档 / 数据 / 设计任务（pdf/xlsx/pptx/docx）最容易被当"纯领域技能任务"而跳过——**切勿豁免**。用 TaskCreate 拆任务时，**最后一项固定加"efw-verify 验收 + efw-learn 沉淀"**，让闸门变成清单里绕不过的一项。
