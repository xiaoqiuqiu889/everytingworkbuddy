# CLAUDE.md — EFCC 编排外壳（Claude 项目级，子代理也读）

> **为什么有这个文件**：用户级 `~/.claude/CLAUDE.md` 的 EFCC 纪律对主对话生效，但**子代理是独立上下文，不加载用户级记忆**。把本文件放进**项目根**，Claude 会在本项目每次会话开头注入它，是让 EFCC 触达"所有在本项目干活的 agent（含子代理）"的唯一可靠通道。
>
> **用法**：复制到你项目根目录，命名 `CLAUDE.md`。删改成你项目的实际约定。

---

## 🍶 本项目采用 EFCC 编排外壳

**EFCC 是套在所有实质任务外面的外壳，不是可选的另一条路。** 无论你是主对话还是被派来的子代理，接到实质任务都走四步：

```
① 计划  →  ② 干活(交对口技能/工具)  →  ③ 验收(efc-verify)  →  ④ 沉淀(efc-learn)
```

**什么算"实质任务"**（满足任一）：① 产出文件/交付物（含 .xlsx/.pptx/.docx/.pdf/.html）；② 需 ≥3 步；③ 预计 ≥15 次工具调用。
→ 纯问答 / 查一条资料 / 单步文本 / 闲聊**不算**，直接做，别套外壳。

> ⛔ **0 号动作**：判定为实质任务 → **第一个动作先用 Skill 工具加载 `efc`**，先于任何领域工具。技能不会自动加载到位，实质任务需你显式发起。
> 🚫 **反"暗合"**：用 TaskCreate 拆任务 ≠ 走外壳；自写校验脚本 ≠ efc-verify 的逐条结构化签收；写 memory 笔记 ≠ efc-learn。神似≠触发。
> 🚦 **DoD 硬闸门**：没跑 `efc-verify`（逐条对照验收标准签收）前，不许说"做完了"。文档/数据/设计任务最容易被当"纯领域任务"跳过——切勿豁免。用 TaskCreate 拆任务时，最后一项固定加"efc-verify 验收 + efc-learn 沉淀"。

**四步怎么走：**
1. **计划**：想清目标、交付物、分步、每步验收标准。对口领域技能自带 plan 的视为满足，不必叠 `efc-plan-feature`。
2. **干活**：交给最对口的领域技能或原生工具。
3. **验收（强制）**：对照第①步标准逐条勾选、给可核验清单，不是肉眼看看就交差。
4. **沉淀**：≥15 工具调用或有可复用套路 → `efc-learn`。

> ⚠️ 最常见漏用：遇到"很能干"的领域技能就把它当 EFCC 的替代、把③④丢了。**领域技能负责干活，EFCC 外壳负责验收+沉淀，两者并存。**

> 📌 **派子代理做实质任务时**：子代理读不到用户级记忆——把上面四步外壳摘要写进它的 prompt，否则它默认不会用 EFCC（本项目已有此 CLAUDE.md 的子代理会自动读到）。
