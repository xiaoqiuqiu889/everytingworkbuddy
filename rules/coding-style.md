# Rule: Coding Style — 编码风格准则

## 核心原则

1. **不可变优先**：默认用不可变数据（`const`/只读结构）；避免共享可变状态；纯函数优先，副作用集中管理。
2. **小而聚焦**：函数单一职责、尽量短；文件不臃肿；一个模块一件事。
3. **命名达意**：名字说清意图，避免缩写和魔法值；用命名常量替代魔法数字。
4. **早返回**：用 guard clause 减少嵌套，避免深层 if/else 金字塔。
5. **显式优于隐式**：明确的错误处理、明确的类型、明确的边界；不吞异常。

## 组织

- 按功能/领域组织目录，而非按技术类型堆放。
- 相关代码就近放置；公共逻辑提取但不过度抽象。
- 遵循项目既有约定 —— 一致性 > 个人偏好。

## 注释

- 注释解释「**为什么**」，不是复述「做了什么」。
- 复杂逻辑、非显而易见的权衡、临时 workaround 才需要注释。
- 过时注释比没注释更糟 —— 改代码时同步改注释。

## 禁忌

- 复制粘贴式重复逻辑。
- 一个函数做太多事。
- 沉默地吞掉错误（空 catch）。
- 为「以后可能用到」而写的抽象（YAGNI）。
