# Rule: Git Workflow — Git 流程准则

## 提交（Commit）

1. **语义化提交信息**：`type(scope): subject`。
   - type：`feat` / `fix` / `refactor` / `docs` / `test` / `chore` / `perf` / `style`。
   - subject 用祈使句、现在时，说清「做了什么」，一行讲明白。
2. **小而聚焦**：一个提交做一件事。不把不相关改动混在一起。
3. **可编译可测试**：每个提交本身应能构建、测试通过（便于二分定位）。
4. **正文说「为什么」**：非显而易见的改动，在正文解释动机与权衡。

## 分支与 PR

- 从主干切功能分支：`feat/xxx`、`fix/xxx`。
- **PR 小而聚焦**：一个 PR 一个目的，便于审查。大改动拆成多个 PR。
- PR 描述：做了什么、为什么、怎么验证、影响范围、回滚方式。
- 合并前：CI 绿、审查通过、无冲突。

## 安全红线

- **不 `--force` 推共享分支**（除非明确协商）。
- **不跳过 hooks**（`--no-verify`）——除非用户明确要求。
- 破坏性操作（`reset --hard`、`clean -fd`）前先确认有备份或可恢复。
- 提交前确认无密钥、无调试代码、无大文件误入。

## 禁忌

- 一个巨型提交塞进一整个功能。
- 提交信息写「update」「fix bug」这类无信息量的内容。
- 直接在主干上开发未经审查就合并。
