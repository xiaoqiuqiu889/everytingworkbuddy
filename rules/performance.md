# Rule: Performance & Context — 性能与上下文准则

## 上下文预算（关键，源自 Everything Claude Code 的核心经验）

> **不要一次性启用所有 MCP/连接器。** 工具太多会大幅挤占上下文窗口，让有效工作空间从 200k 缩到 70k。

经验法则：

- 总共配置 **20–30 个** MCP/连接器即可，不求全。
- **每个项目启用 < 10 个**连接器。
- 保持活动工具数精简（经验值 < 80 个）。
- 用不到的连接器及时关闭 / 不授权。

## 模型选择

- **简单/机械任务**（快速搜索、格式转换、直接文件操作）→ 用轻量模型（lite），省成本、更快。
- **复杂推理**（架构决策、多步问题、疑难调试）→ 用推理增强模型（reasoning）。
- **一般任务** → 默认模型即可。
- 派子代理时按任务难度选模型，不要一律用最贵的。

## 上下文管理

- **广域搜索放子代理**：大量文件读取用 Explore 子代理，只回传结论，避免污染主上下文。
- **及时收敛**：任务完成即总结，不堆积无关中间产物。
- 长会话适时做阶段性小结/压缩，保留关键决策与状态。

## 运行时性能（写代码时）

- 避免 N+1 查询、无谓的重复计算、循环内 IO。
- 大数据集用流式/分页处理，不一次性载入内存。
- 缓存昂贵且稳定的计算结果，但注意失效策略。
- 先测量再优化 —— 不为想象中的瓶颈过早优化。
